# 실패 시 AI 로그 진단·요약 (AI 게이트)

파이프라인의 어떤 단계든 실패하면 `ai-failure-summary` job 이 실패 로그를 수집하고,
Copilot(또는 Foundry/LLM)으로 **원인 → 영향 → 다음 조치**를 3줄로 요약해 PR/Job Summary 에
게시한다. 사람은 긴 로그를 뒤지는 대신 요약과 권고를 먼저 본다.

## 입력
- `gh run view <run_id> --log-failed` 로 추출한 `failed.log` (실패 스텝의 로그만).

## 프롬프트 (하네스가 사용)
```
너는 Database DevOps CI 어시스턴트다. 아래는 실패한 GitHub Actions 잡의 로그다.
목표: DBA/개발자가 즉시 판단할 수 있게 한국어로 3줄 요약하라.

형식:
1) 원인: (가장 가능성 높은 근본 원인 한 줄 — 에러코드/스택 근거 포함)
2) 영향: (배포/데이터/가용성 관점 영향)
3) 다음 조치: (롤백 / 재시도 / 코드수정 중 택1 + 구체 명령·파일)

규칙:
- 추측이면 "가설"이라고 표기. 로그에 없는 사실을 만들지 마라.
- 시크릿/커넥션스트링이 로그에 보이면 값은 마스킹하고 "시크릿 노출 의심"으로 경고하라.
- 마이그레이션 실패면 대응 down 스크립트 경로를 제시하라.

로그:
<<<
{failed.log 내용}
>>>
```

## 출력 예시 (자리표시자)
```
1) 원인: DACPAC publish 중 BlockOnPossibleDataLoss 로 leaderboard 컬럼 축소 차단(SQL72035). (가설 아님)
2) 영향: 배포 롤포워드 중단 — 데이터 손실은 없음(가드가 막음). 스키마 미반영 상태.
3) 다음 조치: expand-contract 로 마이그레이션 분할. 우선 migrations/001_*.down.sql 로 원복 후 재작성.
```

## 배치 위치
- 요약은 `$GITHUB_STEP_SUMMARY` 와 PR 코멘트에 게시(권한: `pull-requests: write`).
- 실제 LLM 호출은 데모 환경/키에 의존하므로 워크플로에는 **훅 스텝**으로만 고정되어 있고,
  키가 없으면 프롬프트·입력·산출 위치를 표시하는 자리표시자로 통과한다(가드).

## 왜 AI 게이트인가 (발표 포인트)
- 실패의 "무엇이/왜"를 사람이 로그에서 찾는 시간을 없앤다.
- 조치를 **롤백/재시도/수정** 중 하나로 못박아 의사결정을 빠르게 한다.
- 값싼 요약은 SLM, 복잡한 원인추적은 LLM — 저장소의 하네스 원칙과 동일.
