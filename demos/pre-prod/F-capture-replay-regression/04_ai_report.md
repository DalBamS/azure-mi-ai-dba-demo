# F — AI 회귀 리포트 템플릿 (04)

`03_compare_waits.sql` 결과를 사람이 읽는 표 대신 **자연어 회귀 리포트**로 요약합니다. 이것이 전통적 DEA(Database Experimentation Assistant) 리포트의 **AI 버전**입니다. 아래 템플릿을 AI 하네스(LLM)에 03 결과와 함께 주면 이런 리포트를 생성합니다.

## 프롬프트 템플릿
> 다음은 baseline 대비 replay 워크로드의 Query Store 비교 결과다(문항별 duration/reads delta 표 + wait-category delta 표). 게임사 DBA가 배포 승인 판단을 하도록: (1) 유의미한 회귀가 있는지, (2) 어떤 쿼리/대기유형이 악화됐는지, (3) 가장 가능성 높은 원인 가설, (4) 배포 진행/보류 권고를 한국어로 요약하라. 근거 수치를 인용하고, 개선된 항목도 균형 있게 언급하라.

## 리포트 형식(생성 결과 예시 골격)
```
## 배포 전 회귀 검증 리포트 — <대상 버전/티어>
- 판정: [진행 가능 / 조건부 / 보류]
- 요약: replay에서 상위 N개 쿼리 중 M개가 회귀(평균 +X ms). 개선 K개.

### 회귀 상위
| query_id | base_ms | replay_ms | Δms | 관찰 |
|----------|---------|-----------|-----|------|
| ...      | ...     | ...       | +.. | 랭킹 조회가 정렬 비용 증가 |

### 대기유형 변화
- 가장 커진 대기: <wait_category> (+X ms) → <해석: 예) 병렬성/락/IO>
- 줄어든 대기: <...>

### 원인 가설
- <예: 통계 갱신 지연으로 카디널리티 추정 악화 / 새 버전의 기본 호환성 레벨 변경>

### 권고
- <예: 문제 쿼리에 대해 Query Store 강제 plan 유지 후 배포 / 특정 인덱스 추가 후 재검증>
```

## 원칙
- 리포트는 **근거(수치) 기반**이어야 하며, 원인은 "가설"로 명시(확정 아님).
- 최종 배포 결정은 **사람 승인**. AI는 근거 수집·요약·가설 제시까지.
