# Pre-prod 데모 (도입 전 검증)

배포/도입 **전에** 워크로드·품질·개인정보 리스크를 미리 잡는 발표용 데모팩입니다. 운영팩과 동일하게 **자연어 입력 → 다단계 자동 진단/생성 → Eval → 사람 승인** 흐름을 따르고, 생성 DB 객체는 대응 rollback/cleanup을 제공합니다.

| 코드 | 데모 | 폴더 | 핵심 |
|------|------|------|------|
| **E** | AI 부하 시나리오 합성 | `E-load-scenario-synthesis/` | 자연어 요구 → 워크로드 프로파일 + 드라이버/HammerDB 파라미터 |
| **F** | 워크로드 캡처 → 리플레이 회귀검증 | `F-capture-replay-regression/` | 티어/버전 간 리플레이 후 wait/duration 자연어 회귀 리포트(DEA의 AI판) |
| **G** | 배포 전 SQL Pre-flight 정적검증 (SLM) | `G-sql-preflight-lint/` | 로컬 Phi-4급 SLM이 SP/쿼리 배치 린팅(누락인덱스·non-SARGable·형변환·풀스캔) |
| **O** | 민감정보 자동분류 + DDM·RLS 정책 생성 | `O-data-classification-masking/` | PII 자동분류 → 마스킹/행수준보안 정책 초안·적용(보안 플래그십) |

## 공통 실행 패턴
1. (해당 시) `00_*`로 데모용 대상/샘플 객체를 준비한다.
2. `01_*`로 근거를 읽기전용 수집하거나(E/F/G/O) 대상을 재현한다.
3. 중간 단계(`02_*`, `03_*`)에서 프로파일/정책/리포트를 **생성(미적용)** 한다 — AI 제안, 사람 검토.
4. `04_eval.sql`(또는 `03_eval.sql`)로 기준·적용 결과를 정량 검증한다.
5. 적용형 데모(O)는 승인 후 적용 스크립트를 실행한다.
6. `05_rollback.sql`/`05_cleanup.sql`로 데모 상태를 정리한다.

## 기존 수동 방식 vs AI 하네스 방식 (공통 프레임)

| 축 | 기존 DBA 수동 방식 | AI 하네스 방식 |
|----|-------------------|----------------|
| 입력 | 요구/시나리오를 사람이 문서로 옮김 | 자연어 한 줄을 그대로 입력(E), 룰셋/정책 근거로 자동화 |
| 생성 | 워크로드·정책·린트를 손으로 작성 | 관측(QS/DMV/분류)을 근거로 결정적 생성 |
| 반복 비용 | 도입 검증마다 사람이 처음부터 | 값싼 반복은 로컬 SLM, 복잡 해석만 LLM |
| 데이터 경계 | 외부 도구에 코드/데이터 반출 우려 | SLM 로컬 실행·MCP 읽기전용으로 경계 유지 |
| 검증 | "괜찮겠지"로 종결 | `*_eval`로 정량 대조(믹스 일치/회귀 delta/안티패턴/카탈로그) |
| 정리 | 잔존 객체·정책 위험 | 대응 rollback/cleanup 제공 |

> 메시지: **도입 전 검증의 반복 노동(부하 설계·회귀 비교·정적 린트·PII 정책 초안)을 하네스가 맡고, 사람은 승인·설계에 집중한다.**

## AI 하네스 메시지
- **값싼 반복 = SLM**(Phi-4 로컬, 예: G 린팅), **복잡 해석 = LLM**(클라우드), **안전 연결 = MCP(읽기전용)**.
- 합성/분류 데이터라 **PII 경계**를 지키며 반복 가능(E는 합성 데이터, O는 격리 데모·합성 PII).
- 실제 Azure 프로비저닝·비밀 하드코딩 없음. `.env`/환경변수/Key Vault 참조만.
